Setup guide

Get a local review loop running in under a minute.

Requirements

  • Python 3.8+ on your PATH (python3, python, or py -3).
  • git on your PATH.
  • No third-party packages for the server or skill. The shell loads highlight.js / marked / DOMPurify / diff2html from a CDN, so a browser with internet access renders best.
The skill commands are plain Python (stdlib only), so they run the same on Windows, macOS, and Linux.

Quick start (via the skill)

  1. Launch the bridge from the repo you want to review: 
    python3 path/to/agentic-review/skills/agentic-review/scripts/launch.py
    It starts the loopback server and prints a ready-to-open URL.
  2. Open the review shell. Use the printed link, e.g. 
    http://127.0.0.1:8900/review.html?token=…
    Review the diff (or browse all files) and click a line or block to leave comments.
  3. Feed comments back to the agent: 
    python3 .../scripts/take-feedback.py
  4. Tear down when you are done: 
    python3 .../scripts/cleanup.py --force

Run the bridge directly

You can skip the skill and start the server yourself:

python3 local-server/server.py --root /path/to/repo --port 8900
# same-origin shell:  http://127.0.0.1:8900/review.html
# landing / docs:     http://127.0.0.1:8900/

The bridge binds to 127.0.0.1 only and confines file access to the repo root (path traversal, absolute paths, and symlink escape are rejected).

What counts as "under review" (the diff base)

By default the review shows uncommitted changes (working tree vs HEAD), so once a change is committed it drops out of the review — the shell auto-refreshes and tells you when that happens. To review committed history instead, set the diff base to an earlier ref: 

# review the last commit
AR_DIFF_BASE=HEAD~1 python3 .../scripts/launch.py

# diff against main
python3 local-server/server.py --root . --diff-base main --port 8900

To make every tracked file show up (e.g. to try the renderers in examples/), point the diff base at the empty tree: 

python3 local-server/server.py --root "$PWD" \
  --diff-base "$(git hash-object -t tree /dev/null)" --port 8900

Deploying the shell

The shell is fully static. Two ways to serve it:

1. Same-origin (default, zero-config)

The bridge serves site/ itself, so the review shell is at http://127.0.0.1:<port>/review.html and this docs site is at http://127.0.0.1:<port>/. There is no cross-origin call at all.

2. GitHub Pages

The shell is published at wangxi-dev.github.io/agentic-review/review.html (via .github/workflows/static.yml, source = "GitHub Actions"). The bridge always allows this origin, so just open: 

https://wangxi-dev.github.io/agentic-review/review.html?api=http://localhost:<port>&token=…

agentic-review:launch prints this link for you. The bridge sends Access-Control-Allow-Private-Network: true on preflights for Chrome's Private Network Access.

For any other static host, launch the bridge with AR_PAGES_ORIGIN=https://your.host to add that origin to the CORS allowlist (and AR_PAGES_URL=https://your.host/path/ to make launch print its link).

Browser support: an HTTPS Pages shell calling http://localhost works in current Chrome, Firefox, and Safari via the loopback exception. If a browser blocks the localhost call, use the same-origin option above.

Checks (pluggable code checkers)

The shell can run code checkers on the open file ("Run checks" in the viewer header). Built-ins:

  • Lines of code — flags files over 800 lines and lines over 250 characters.
  • Code complexity — flags nesting deeper than 4 and functions with more than 4 parameters.

Drop your own executable CLI into <repo>/.agentic-review/checkers/. A checker must support:

Full contract, a copy-paste example, and the built-in thresholds are in local-server/checkers/README.md in the repo. 
checker --describe        # prints {"id","name","description"} as JSON
checker <relative-path>   # reads file CONTENT on stdin, prints findings JSON

Findings format (stdout):

{ "findings": [
  { "line": 12, "severity": "warning", "rule": "max-line-length",
    "message": "Line is 312 chars (> 250)" },
  { "severity": "error", "rule": "max-file-loc",
    "message": "File has 950 lines (> 800)" }
] }
Security: checkers are executables run on your machine on demand. Only place checkers you trust in .agentic-review/checkers/. That folder is git-ignored and not populated by cloning a repo, so a reviewed repo cannot inject checkers.

Tests

python3 local-server/test_server.py   # stdlib unittest: unit + end-to-end

Typical round trip

  1. launch → open the URL, review the diff (or all files) and comment.
  2. take-feedback → read comments, make the requested changes.
  3. (optional) precommit → stage the commit message and have the human review it.
  4. cleanup --force once satisfied; then commit.
